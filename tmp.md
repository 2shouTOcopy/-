一、我对当前澄清后需求的整体理解

这个需求现在可以理解为：

FTP 文本传输能力从“仅支持 TXT、按分钟生成新文件”扩展为：
	1.	支持 TXT / CSV / JSON 三种文本格式；
	2.	支持为内容增加时间戳字段；
	3.	支持将用户订阅的文本按 分号 ; 拆分后，组织为 CSV 列或 JSON 项；
	4.	支持固定文件名覆盖更新，文件名由用户配置，默认 result；
	5.	支持缓存并维护最近 N 条结果，并按一定条件批量刷新上传；
	6.	在停流等结束场景下，允许做一次最终刷新，避免尾部结果丢失。

二、已明确的需求拆解

⸻

1. 新增时间戳设置按钮

我的理解

新增一个开关参数，控制导出内容中是否携带时间戳。

建议定义

参数名建议：
	•	是否携带时间戳
	•	或 时间戳使能

类型：
	•	bool

默认值建议：
	•	关闭

不同格式下的表现建议

TXT
如果开启时间戳：

2026-04-07 10:00:00;content1;content2;content3

或者：

[2026-04-07 10:00:00] content1;content2;content3

我更建议第一种，结构更统一，也方便以后兼容 CSV/JSON。

CSV
时间戳作为第一列：

timestamp,col1,col2,col3
2026-04-07 10:00:00,content1,content2,content3

JSON
时间戳作为一个字段：

[
  {
    "timestamp": "2026-04-07 10:00:00",
    "item1": "content1",
    "item2": "content2",
    "item3": "content3"
  }
]


⸻

2. CSV 和 JSON 按分号拆分

我的理解

用户订阅得到的是一整段文本，例如：

content1;content2;content3

系统收到后，以 ; 为分隔符进行拆分：
	•	第 1 段 → CSV 第一列 / JSON 第一项
	•	第 2 段 → CSV 第二列 / JSON 第二项
	•	第 3 段 → CSV 第三列 / JSON 第三项

这意味着

CSV
一条结果对应一行：

col1,col2,col3
content1,content2,content3

如果带时间戳：

timestamp,col1,col2,col3
2026-04-07 10:00:00,content1,content2,content3

JSON
一条结果对应一个对象：

[
  {
    "item1": "content1",
    "item2": "content2",
    "item3": "content3"
  }
]

如果带时间戳：

[
  {
    "timestamp": "2026-04-07 10:00:00",
    "item1": "content1",
    "item2": "content2",
    "item3": "content3"
  }
]


⸻

3. 保留最近 N 条结果 / 保留最近时间的结果

这一条是整个需求的核心。

我的理解

系统不再是来一条就发一条，也不再是每分钟新建文件，而是要维护一个缓存窗口：
	•	始终保留最近 N 条结果
	•	或从时间维度上保留最近一段时间内的结果
	•	FTP 文件内容始终反映这个“最近结果窗口”
	•	每隔一定数量的新增结果，再触发一次覆盖上传
	•	停流事件发生时，再补刷新一次

这个其实已经不是单纯“文件格式扩展”，而是新增了结果缓存 + 刷新策略。

⸻

这里我建议把需求在文档里明确拆成两个概念

A. 保留策略
决定“缓存里存什么”

候选：
	•	保留最近 N 条结果
	•	保留最近 T 时间内的结果

B. 刷新策略
决定“什么时候重新上传覆盖文件”

候选：
	•	每新增 M 条结果刷新一次
	•	停流时补刷新一次

这样逻辑就清楚了。

⸻

4. 文件名开放节点供用户配置，默认 result

我的理解

覆盖上传时使用固定文件名，不再使用 %Y_%m_%d_%H_%M.xxx 这类按时间生成的文件名。

建议定义

新增节点：
	•	文件名

类型：
	•	string

默认值：
	•	result

最终文件名由：
	•	用户文件名 + 格式后缀

组成，例如：
	•	result.txt
	•	result.csv
	•	result.json

这里建议补一个约束

需要在文档中约束：
	•	用户配置文件名时不需要带后缀
	•	系统根据当前格式自动补后缀

否则用户可能填：
	•	result.csv

然后切换成 JSON 时会出现：
	•	result.csv.json

这种不一致问题。

三、建议你补充到 OR 文档中的“可落地需求定义”

下面这版可以直接拿去改写进文档。

⸻

需求点1：新增文本文件格式类型

FTP 文本传输新增支持以下格式：
	•	TXT
	•	CSV
	•	JSON

默认格式为：
	•	TXT

⸻

需求点2：新增时间戳使能参数

新增参数：
	•	时间戳使能

功能说明：
	•	关闭时，仅输出用户订阅内容
	•	开启时，在每条结果中增加时间戳信息

不同文件格式下的表现：
	•	TXT：时间戳作为每条文本记录前缀
	•	CSV：时间戳作为第一列
	•	JSON：时间戳作为对象中的字段

⸻

需求点3：CSV/JSON 内容组装规则

对用户订阅得到的文本内容，以分号 ; 作为分隔符进行拆分。

例如订阅文本为：

content1;content2;content3

则：

CSV 组装方式
	•	每条结果对应一行
	•	分号拆分后的各段分别映射为不同列
	•	列名默认使用 col1、col2、col3 …
	•	若启用时间戳，则时间戳列名固定为 timestamp

JSON 组装方式
	•	每条结果对应一个对象
	•	分号拆分后的各段分别映射为不同字段
	•	字段名默认使用 item1、item2、item3 …
	•	若启用时间戳，则增加 timestamp 字段

⸻

需求点4：新增结果缓存保留策略

系统支持缓存最近结果，并基于缓存内容生成 FTP 文件。

支持的保留策略建议明确为：
	•	按最近 N 条结果保留
	•	按最近时间范围保留

缓存存在上限限制，用于避免内存无限增长。
例如上限为 10000 条时，仅保留最新的 10000 条结果，超出部分按先进先出淘汰。

⸻

需求点5：新增刷新上传策略

系统支持按条件对 FTP 文件进行覆盖上传。

建议支持：
	•	每累计新增 M 条结果，执行一次覆盖上传
	•	在停流事件中执行一次最终刷新上传

例如：
	•	缓存上限：10000 条
	•	刷新步长：100 条

则每新增 100 条结果时，对 FTP 文件执行一次覆盖更新。
FTP 文件内容为当前缓存窗口中的结果集合。

⸻

需求点6：新增自定义文件名

新增参数：
	•	文件名

默认值：
	•	result

规则：
	•	用户仅配置主文件名，不包含扩展名
	•	系统根据当前文件格式自动追加后缀

例如：
	•	TXT → result.txt
	•	CSV → result.csv
	•	JSON → result.json

四、我建议你继续和产品确认的边界点

虽然主需求已经清楚，但还有几个边界最好继续定掉，否则后面实现时容易各自理解不一致。

⸻

1. “保留最近时间的结果”具体是什么意思？

这个表述目前还不够严谨。

有两种可能：

理解A：保留最近 T 秒 / T 分钟内产生的所有结果

例如保留最近 10 分钟结果。

理解B：保留最近某个时间点之后的结果

这种不太像常规产品表达。

我建议你和产品明确成：
	•	按时间窗口保留
	•	例如：最近 10 分钟 / 最近 1 小时

否则研发实现会有歧义。

⸻

2. 刷新步长和缓存上限是不是两个独立参数？

你举的例子是：
	•	上限 10000 条
	•	每隔 100 条发送一次

这说明实际上是两个参数：
	•	缓存上限
	•	刷新步长

建议明确独立出来，不要混成一个概念。

否则会出现问题：
	•	用户想保留 10000 条，但每 1 条都刷新
	•	用户想保留 500 条，但每 50 条刷新

这是两种不同诉求。

⸻

3. CSV 列数 / JSON 项数不一致时怎么处理？

比如某几条内容是：

a;b;c

另一条是：

a;b

再一条是：

a;b;c;d

这时必须定义规则。

我建议默认规则

CSV
以当前缓存中的最大列数为准，不足列补空。

例如：

col1,col2,col3,col4
a,b,c,
a,b,,
a,b,c,d

JSON
按实际拆分数量生成字段即可，不足的字段可以不生成，也可以补空。
我更建议按实际字段数生成，更自然。

这点最好在文档中写清。

⸻

4. TXT 在新机制下的表现要不要也按“最近 N 条”组织？

现在 CSV/JSON 已经是“结构化缓存输出”了。
TXT 最好也统一逻辑，否则三种格式行为会不一致。

我建议 TXT 也这样定义：
	•	缓存中的每条结果占一行
	•	若开启时间戳，则每行前带时间戳
	•	文件整体内容为当前缓存窗口中的所有记录

这样最一致。

⸻

5. 停流刷新时的行为要不要受“本轮是否有新增结果”限制？

建议明确：
	•	仅当上次发送后存在未刷新的新增结果时，停流事件才触发最终刷新
	•	否则不重复发送

这样可以避免无意义重复上传。

六、研发实现上我建议重点注意的地方

⸻

1. 缓存容器建议做成环形队列或 deque

因为需求是“保留最近 N 条”，本质上是滑动窗口。

适合用：
	•	std::deque<Record>
	•	或固定容量环形缓冲区

这样插入和淘汰都方便。

⸻

2. “缓存上限”和“刷新步长”不要耦合

这是两个不同概念：
	•	缓存上限：控制保留多少历史
	•	刷新步长：控制上传频率

内部最好分开配置。

⸻

3. 文件覆盖上传最好考虑原子性

如果 FTP 服务端和当前实现允许，最好考虑：
	•	先上传临时文件
	•	再 rename 成正式文件

否则下游正好读取时，可能读到半截内容。

如果当前 FTP 库不方便做 rename，至少要在文档里说明当前是直接覆盖上传。

⸻

4. CSV/JSON 格式化逻辑独立封装

建议不要把拼字符串逻辑散在 FTP 流程里。

可以单独做：
	•	FormatTxt(records, withTimestamp)
	•	FormatCsv(records, withTimestamp)
	•	FormatJson(records, withTimestamp)

这样后续更容易扩展。
